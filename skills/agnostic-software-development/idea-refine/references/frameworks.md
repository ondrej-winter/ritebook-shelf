# Ideation Frameworks Reference

Use these frameworks selectively. The goal is to unlock better thinking, not to
complete a checklist.

## SCAMPER

Use SCAMPER to transform an existing idea through seven operations:

- substitute: swap the audience, technology, channel, business model, or process
- combine: merge the idea with another product, service, workflow, or domain
- adapt: borrow a pattern from another industry, time period, or natural system
- modify: make one dimension much larger, smaller, faster, slower, richer, or
  simpler
- put to other uses: apply the idea to a different user, context, or problem
- eliminate: remove a feature, step, dependency, approval, or configuration point
- reverse or rearrange: invert the sequence, value chain, ownership model, or user
  responsibility

Best for improving or reimagining an existing product, feature, or workflow.

## How Might We

Use a How Might We frame to convert a pain point into an open problem statement:

```text
How might we <desired outcome> for <specific user> without <key constraint>?
```

A good framing is narrow enough to act on, broad enough to allow multiple
solutions, and clear about the tension that makes the problem interesting.

Avoid framings that are:

- too broad, such as making users happy
- too narrow, such as adding a specific button
- solution-embedded, such as building a chatbot before support needs are clear

Best for pulling the conversation back from a proposed solution to the underlying
problem.

## First Principles Thinking

Break the idea down to what is actually true, then rebuild from there:

1. List what is known, not assumed.
2. List every meaningful assumption.
3. Ask which assumptions are conventions rather than real constraints.
4. Rebuild the concept from the remaining truths.

Best for escaping incremental thinking or inherited constraints.

## Jobs to Be Done

Focus on the job the user is hiring the idea to perform:

- functional job: what task are they trying to complete?
- emotional job: how do they want to feel?
- social job: how do they want to be perceived?

Use this format:

```text
When I <situation>, I want to <motivation>, so I can <expected outcome>.
```

Best for discovering whether the proposed solution addresses the real problem.

## Constraint-Based Ideation

Add constraints deliberately to force sharper options:

- time: what if this had to ship in one day?
- feature count: what if it could have only one feature?
- technology: what if the obvious technology were unavailable?
- cost: what if it had to be free to operate?
- audience: what if the user had no training or no patience?
- scale: what changes for ten users? What changes for a billion?

Best for cutting through vague or overgrown concepts.

## Pre-Mortem

Imagine the idea failed after launch, then work backward:

1. Name every plausible reason it failed.
2. Separate technical, market, team, timing, and trust failures.
3. Decide which failures are preventable.
4. Decide which failures would kill the idea.

Best for convergence and assumption auditing when an idea feels good but has not
faced hard questions.

## Analogous Inspiration

Look for structural similarities in other domains:

- Which industry already solved a similar coordination, trust, discovery, or
  adoption problem?
- What would this look like in a different medium, business model, or community?
- What natural, historical, or operational system has the same shape?

Avoid shallow analogies. The useful analogy is the underlying structure, not the
surface comparison.

Best for generating directions that are meaningfully different from the obvious
one.
